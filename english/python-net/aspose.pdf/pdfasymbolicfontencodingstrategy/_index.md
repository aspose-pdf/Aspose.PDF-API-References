## PdfASymbolicFontEncodingStrategy class

This class describes rules which can be used to tune process of copy encoding data for cases<br/>            when TrueType symbolic font has more than one encoding.<br/>            Some PDF documents after conversion into PDF/A format could have error <br/>            "More than one encoding in symbolic TrueType font's cmap". <br/>            What is a reason of this error? All TrueType symbolic fonts have special table "cmap"<br/>            in it's internal data. This table maps character codes to glyph indices. <br/>            And this table could contain different encoding subtables which<br/>            describe encodings used. See advanced info about cmap tables at<br/>            https://developer.apple.com/fonts/TrueType-Reference-Manual/RM06/Chap6cmap.html.<br/>            Usually cmap table contains several encoding subtables, but PDF/A standard requires<br/>            that only one encoding subtable (3,0) must be leaved for this font in PDF/A document. <br/>            And key question here - what data must be taken from another subtables to copy into<br/>            destination encoding table (3,0)? Majority of fonts have 'well-formed' cmap tables where<br/>            every encoding subtable is fully consistent with another subtable. But some fonts<br/>            have cmap tables with collisions - where for example one subtable has glyph index <br/>            100 for unicode 100, but another subtable has glyph index 200 for the same unicode 100.<br/>            To solve this problems special strategy needed. <br/>            By default following strategy used: <br/>            mac subtable(1,0) is looked for. If this table is found, only this data used to fill destination<br/>            table (3,0). If mac subtable is not found then all subtables except (3,0) are iterated <br/>            and used to copy data into destination (3,0) subtable. Also mapping for every unicode(unicode, glyph index)<br/>            is copied into destination table only if destination table has no this unicode at current moment. <br/>            So, for example if first subtabe has glyph index 100 for unicode 100, and next subtable has glyph<br/>            index 200 for the same unicode 100, only data from first subtable (unicode=100, glyph index = 100) will be copied. <br/>            So each previous subtable takes precedence over the next. <br/>            Properties of this class [PdfASymbolicFontEncodingStrategy](/pdf/python-net/aspose.pdf/pdfasymbolicfontencodingstrategy/) help tune default behaviour. <br/>            If property [preferred_cmap_encoding_table](/pdf/python-net/aspose.pdf/pdfasymbolicfontencodingstrategy/) of type [CMapEncodingTableType](/pdf/python-net/aspose.pdf.pdfasymbolicfontencodingstrategy.queueitem/cmapencodingtabletype/)<br/>            is set, then relevant subtable will be used in precedence to mac subtable(1,0). Value 'MacTable' from<br/>            enumeration [CMapEncodingTableType](/pdf/python-net/aspose.pdf.pdfasymbolicfontencodingstrategy.queueitem/cmapencodingtabletype/) has no sense in this case, cause it <br/>            points on the same mac subtable (1,0) which will be used by default. <br/>            Property [None](/pdf/python-net/aspose.pdf/pdfasymbolicfontencodingstrategy/) discards all priorities for any subtable.<br/>            If this property is set, then only subtables from declared queue will be used in specified order.<br/>            If subtables specified are not found then default iteration of all subtables and copy strategy described above<br/>            will be used.<br/>            Object [QueueItem](/pdf/python-net/aspose.pdf.pdfasymbolicfontencodingstrategy/queueitem/) specifies encoding subtable used. This subtable can be set<br/>            via combination of members(PlatformID, PlatformSpecificId) or via [CMapEncodingTableType](/pdf/python-net/aspose.pdf.pdfasymbolicfontencodingstrategy.queueitem/cmapencodingtabletype/)<br/>            enumeration.
